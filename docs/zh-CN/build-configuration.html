Configuration

bsconfig 是 bsb 所需的强制生成的单个元文件。

完整的配置描述 在这里。 我们将 非详尽的 介绍一下最重要的部分。

name, namespace

name 是库的名称, 也用来作为其 "命名空间"。 您可以在bsconfig.json中通过 "namespace": true 来激活命名空间。 Namespacing 几乎是 强制的， 默认情况下, 我们还没有打开它以保留向后兼容性。

解释: 默认情况下, 您的文件 (一旦用作第三方依赖项) 可在全球范围内提供给使用者。 例如, 如果您有 Util , 并且使用者也有同名的文件, 它们就会发生冲突。 打开 命名空间 可通过将所有项目的文件包装到额外的模块层来避免此问题。使用者将看到你的包为 MyProject. Util，而不是全局 Util 模块。 namespacing 影响的是使用者, 而不是您自己的。

在 Bsb 中, "命名空间" 只是一个自动生成的模块的一个花哨术语, 它将所有项目的文件 (当然是有效的和正确的) 包装在命名空间里，以能被第三方引用而不导致冲突。

我们不为您自己的项目做文件夹级命名空间; 所有您自己的文件名必须是唯一的。 这是一个约束, 它支持一些功能, 如快速搜索和更容易的项目重组。

注释: bsconfig. json中 name 应与 package.json 中name相同, 以避免混淆。 但是, 这意味着您不能使用 camelCased 驼峰式命名 (如 MyProject), 因为 package. json 和 npm 禁止您这样做 (某些文件系统不区分大小写)。 要将命名空间/模块作为 MyProject, 请配置 "name": "my-project"。 Bsb 将正确地将其转化为 camelCased 驼峰式名称。

Note on custom namespacing: if for some reason, you need a namespace that is different from what your name will produce, you can directly send a string to the namespace option. For example, if your package is a binding named bs-some-thing, you can use "namespace": "some-thing" to get SomeThing namespace instead of BsSomeThing.

sources

Your source files need to be specified explicitly (we don't want to accidentally drill down into some unrelated directories). Examples:

{
  "sources": ["src", "examples"]
}

{
  "sources": {
    "dir": "src",
    "subdirs": ["page"]
  }
}

{
  "sources": [
    "examples",
    {
      "dir": "src",
      "subdirs": true // recursively builds every subdirectory
    }
  ]
}

You can mark your directories as dev-only (for e.g. tests). These won't be built and exposed to third-parties, or even to other "dev" directories in the same project:

{
  "sources" : {
    "dir" : "test",
    "type" : "dev"
  }
}

bs-dependencies, bs-dev-dependencies

List of BuckleScript/Reason dependencies. Just like package.json's dependencies, they'll be searched in node_modules.

Note that only sources marked with "type":"dev" will be able to resolve modules from bs-dev-dependencies.

reason, refmt

reason config is enabled by default. To turn on JSX for ReasonReact, specify:

{
  "reason": {"react-jsx": 3},
  "refmt": 3
}

The refmt config should be explicitly specified as 3, as in the Reason V3 syntax.

js-post-build

Hook that's invoked every time a file is recompiled. Good for JS build system interop, but please use it sparingly. Calling your custom command for every recompiled file slows down your build and worsens the building experience for even third-party users of your lib.

Example:

{
  "js-post-build": {
    "cmd": "/path/to/node ../../postProcessTheFile.js"
  }
}

Note that the command's path resolution is done through the following:

  • /myCommand解析成 /myCommand
  • myCommand/解析成 node_modules/myCommand
  • ./myCommand解析成 myProjectRoot/myCommand
  • myCommand 被当作 myCommand执行. 但是请注意, Bsb 没有读取您的环境, 因此, 如果您放到 node中运行 , 它不会找到它, 除非您指定绝对路径。 或者, 在脚本头部进行标明 #!/usr/local/bin/node 。

The command itself is called from inside lib/bs.

package-specs

Output to either CommonJS (the default), ES6 modules or AMD. Example:

{
  "package-specs": {
    "module": "commonjs",
    "in-source": true
  }
}
  • "module": "es6-global" 使用相对路径解析 node_modules 模块。 好在是浏览器已经支持在开发时用的ES6的用法, 如 Safari 和火狐已经支持 ES6 模块了。 不再需要开发时打包了！!
  • "in-source": true与源文件一起生成输出。如果省略它, 它将生成的代码放在 lib/js 下面。如果没有额外配置输出目录的话。

This configuration only applies to you, when you develop the project. When the project is used as a third-party library, the consumer's own bsconfig.json package-specs overrides the configuration here, logically.

suffix

Either ".js" or ".bs.js". Strongly prefer the latter.

设计决策

Generating JS files with the .bs.js suffix means that, on the JS side, you can do const myBuckleScriptFile = require('./theFile.bs'). The benefits:

  • 很明显, 我们正在处理的是一个生成的 JS 文件。
  • 它避免与 theFile.js 同一文件夹中的潜在文件发生冲突。
  • 它避免了需要使用一个构建的loader去处理 BuckleScript 文件。 in-source 源代码生成意味着可以将 BuckleScript 项目集成到您的纯 JS 代码库中，基本上不用改动任何构建管道中的任何内容。
  • bs. js 后缀 可以让 bsb 更好的跟踪 js 生成代码。

warnings

Selectively turn on/off certain warnings and/or turn them into hard errors. Example:

{
  "warnings": {
    "number": "-44-102",
    "error": "+5"
  }
}

Turn off warning 44 and 102 (polymorphic comparison). Turn warning 5 (partial application whose result has function type and is ignored) into a hard error.

The warning number are shown in the build output when they're triggered. The complete list is here, a little bit below. 100 and up are BuckleScript-specific.

bsc-flags

Extra flags to pass to the underlying bsc call. Notably: ["-bs-super-errors"] for turning on cleaner error output. No need to pass this flag for a Reason project; it's enabled by default.

Environment Variables

We heavily disrecommend the usage of environment variables, but for certain cases, they're justified.

Error Output Coloring: NINJA_ANSI_FORCED

This is mostly for other programmatic usage of bsb where outputting colors is not desired.

When NINJA_ANSI_FORCED is set to 1: bsb produces color. When NINJA_ANSI_FORCED is set to 0: bsb doesn't produce color. When NINJA_ANSI_FORCED is not set: bsb might or might not produce color, depending on a smart detection of where it's outputted.

Note that bsc, the barebone compiler, will always be passed -color always. See more details in this issue.
