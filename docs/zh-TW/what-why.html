What & Why

什麼是 BuckleScript ？

BuckleScript 並不是一個新的程式語言。 它單純使用了 OCaml ：一個快速、程序性、強型別的語言，來編譯成乾淨、可讀、且有效率的 JavaScript 程式。 這使得使用者能夠既寫出業界強度等級的語言且同時能使用現存熟悉的 JavaScript 工具 (npm/yarn, bundlers, minifiers, devtools, 等等) ，並且可以同時使用 JavaScript 和 OCaml 的廣大生態系。

由於它與 JS 的互通性，BuckleScript 可以被視為一種「逐漸」型別化的語言，你可以一個一個檔案逐步採用它，且同時得到 100% 可靠的型別系統的好處與額外的語言功能。

BuckleScript 內建支援 Reason (我們的合作夥伴專案) ，以便為現有 JS 開發人員帶來更熟悉的體驗。

為什麼需要編譯成 JavaScript ？

JavaScript 的生態系統和社群非常活躍。 這個語言本身在採用與分佈方面是無與倫比的。 因此編譯成 JavaScript 是一個務實的選擇（有時也是唯一的選擇）。

此外，JavaScript 虛擬機器還經過了許多工程年的微調而變得更快。

為何用 BuckleScript ？

「為什麼不用 JavaScript 就好了？」

儘管 JavaScript 擁有明顯的優勢，它仍存在一些缺點。以下是 BuckleScript 解決這些缺點的方法。

可靠性

BuckleScript 的型別系統本身提供了三大能夠讓你的程式更可靠的好處：

  • 健全的型別系統： 「健全」在這裡代表著 保證 型別一定如此，不會只有 90% 的時間符合。 一旦 BuckleScript 的專案編譯之後，就不會有運行時的型別錯誤。 *
  • 強型別推斷： 幾乎整個語言都可以被推斷出來。 你不需要繁瑣地手動寫出所有的型別。 但你仍然能夠為了可讀性而寫出一部分的型別！
  • 具有表達力的型別功能： 這些型別引導您完成反覆的開發過程且不會阻礙您表達所需的內容，因為它們具有經過了深思熟慮而開發出的功能：例如 變體 (variant)、 模組 (module) ，甚至可以選擇加進 物件系統 (object system)。

結合這三點以及其他型別系統的功能，開啟了新的工作流的可能性，而且替 JavaScript 提供了它所沒有的漸進式型別化解決方案。

* 有些蓄意的、可選擇的互通性掛鉤 (hooks) 的例外情況

高品質的無用程式碼移除

JavaScript 生態系很多時候都非常依賴套件 (dependency) 。 產出最後的產品時，通常無可避免地挾帶了大量但實際沒有使用到的程式碼。 這些無用的程式碼影響了載入、解析和解譯的速度。 BuckleScript 提供了強大的無用程式碼移除功能。

  • 精心設計的型別系統和 純度分析 (purity analysis) 促進了函數和模組等級的程式碼移除
  • 在全域等級方面，BuckleScript 在經過了它自己的精密移除流程後，利用打包工具，如：Rollup 和 Closure Compiler ，來產生自然友善移除了無用程式碼的程式碼。
  • 相同的事情也應用於 BuckleScript 自己本身的（也是由 BuckleScript 寫成的）小執行期 (runtime) 。

編譯時的優化

JavaScript 是個動態的程式語言，它在瀏覽器執行期優化程式碼的時候會受到性能影響。 雖然有些瀏覽器引擎在一定程度上利用 暫存 (caching) 解決了這個問題，但這些並不是一定可用或是可被預期的。 程式碼能夠被優化的程度也因為沒有內建的強型別系統而受限。 然而， BuckleScript 能夠遠在程式碼被發送到瀏覽器之前，意即在編譯的時候，就提供許多優化。 廣泛的共識來說，寫快速的 JavaScript 程式碼的方式是假設它有型別系統（為了觸發 JS 引擎的所擁有的優化）。BuckleScript 提供一個真的型別系統並且預設產生了友善於優化的程式碼。

也可以在原生 (Native) 執行！

BuckleScript 真的只是 OCaml（除了對於 JavaScript 的交互操作 (interop) ）。 OCaml 可以直接編譯成機器碼 (machine code) ，意即快速的準組合碼 (assembly) *。 如果你沒有針對 JavaScript ，你可以對伺服器、桌面、行動裝置的作業系統重複使用一樣的程式碼。 如此一來提供了更好的效能。 舉例來說，傳統的 node.js 應用程式需要花 >100ms 來啟動，但經過 OCaml 編譯過的機器碼只需要大約 2ms 的時間。

* 事實上，它更可以只用幾 KBs 的記憶體跑在 微處理機 (microcontrollers) 上。

BuckleScript 有什麼不同之處？

現在已經有很多語言能夠被編譯成 JavaScript。BuckleScript 與這些替代選擇有以下幾點關鍵的不同之處：

快速的迭代

BuckleScript 的建制生成時間比其他選擇還快一到兩個數量級。 在它的觀察模式下，生成系統通常在你從編輯器切換到終端機的分頁（約幾十毫秒）之前就完成了。 快速的迭代週期使得我們可以減少花太久時間來集中精神狀態的情形；這使得人們可以更常長時間的駐足在整個流程上。

可讀的輸出與優異的交互操作 (Interop)

其他能夠編譯成 js 的語言所產生的無法閱讀的程式碼可能會造成以下的困難：

  • 難以調試除錯（隱藏的堆疊追蹤、亂掉的變數名稱）
  • 難以學習（沒有直接從一種語言到另一個語言在概念上的對應關係）
  • 難以分析性能（不清楚運行時性能成本有多少）
  • 難以與現有的 JS 程式碼整合

BuckleScript 的 JS 輸出具有良好的可讀性。這對於學習尤其重要，因為使用者可能會希望理解程式碼的編譯方式以及如何審核 bug。

這種特性與功能完備的 JS 交互操作 (interop) 系統結合，使得 BuckleScript 程式碼能夠幾乎不被注意到的被插入到現有的 JavaScript 程式碼庫裡面。

微小的 JS 輸出

一個 Hello world 的 BuckleScript 程式產生約 20 bytes 的 JS 程式碼。此外，你所需要的標準函式庫的功能只會在需要的時候被包含在裡面。

保留了程式碼的架構

BuckleScript 會對應於每個 OCaml/Reason 檔案產生一個 JavaScript 檔案。 這使得對於現有工具（如 bundlers 與 test runners）的整合更加簡單。 您甚至可以無需更改生成設置 (build setup) 就直接開始編寫檔案。 而且每個檔案的程式碼架構幾乎都能夠被保留下來。
