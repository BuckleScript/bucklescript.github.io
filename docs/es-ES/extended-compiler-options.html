Extended Compiler Options

Nota: esta sección es sólo para personas que quieren hacer su propio sistema de construcción en vez de usar el sistema de construcción recomendado, bsb. Esto adicionalmente provee algunos consejos para contribuyentes que depuran el compilador.

BuckleScript hereda los argumentos de linea de comando del compilador OCaml. Además añade varias banderas específicas de BS. Ejecuta bsc -help para ver la lista.

Estas banderas pueden pasarse a la sección bsc-flags de bsb (aunque no tiene sentido pasar algunas de ellas).

Nota además que esta sección no se mantiene muy actualizada. ¡Las contribuciones son bienvenidas!

-bs-main (construcción de directorio único)

principal -bs-main bsc

bsc construirá el módulo Main con todas sus dependencias. Cuando termine, correrá node main.js.

principal -c -bs-main bsc

Igual al anterior, pero no ejecutaará node.

-bs-files

Así que puedes hacer

bsc -c -bs-files *.ml *.mli

el compilador ordenará los archivos de entrada antes de empezar la compilación.

BuckleScript soporta dos modos de compilación: modo script y modo paquete. En modo paquete, tienes que proveer package.json arriba y configurar las opciones -bs-package-name, -bs-package-output. En modo script, esas banderas no son necesarias.

-bs-package-name

El nombre de tu proyecto. Es recomendado usar el mismo name que el usado en package.json.

-bs-package-output

Configura el sistema de módulo de salida. El formato es module_system:output/path/relative/to/package.json.

Actualmente los sistemas soportados son: commonjs, amdjs y es6.

Por ejemplo, cuando quieres usar el sistema de módulo es6:

bsc -bs-package-name your_package -bs-package-output es6:lib/es6 -c xx.ml

Nota: puedes proveer multiples -bs-package-output a la vez. Por ejemplo:

bsc -bs-package-name name -bs-package-output commonjs:lib/js  -bs-package-output amdjs:lib/amdjs -c x.ml

Esto generará x.js en lib/js como un módulo commonjs, lib/amdjs como un módulo amdjs al mismo tiempo.

Ahí entonces necesitarías un bundler para los diferentes sistemas de módulos: webpack soporta commonjs y amdjs, rollup soporta es6, mientras que el compilador google closure los soporta todos.

-bs-no-warn-ffi-type

Turn off warnings on FFI type declarations.

-bs-eval

Example:

> bsc -dparsetree -drawlambda -bs-eval 'Js.log "hola"'

[
  structure_item (//toplevel//[1,0+0]..[1,0+14])
    Pstr_eval
    expression (//toplevel//[1,0+0]..[1,0+14])
      Pexp_apply
      expression (//toplevel//[1,0+0]..[1,0+6])
        Pexp_ident "Js.log" (//toplevel//[1,0+0]..[1,0+6])
      [
        <label> ""
          expression (//toplevel//[1,0+7]..[1,0+14])
            Pexp_constant Const_string("hola",None)
      ]
]

(setglobal Bs_internal_eval! (seq (log "hola") (makeblock 0)))
// Generado por BUCKLESCRIPT VERSIÓN 2.1.0, POR FAVOR EDITE CON CUIDADO
'use strict';


console.log("hola");

/*  No es un módulo puro */

In conjunction with -bs-eval: the first block is the output of -dparsetree, the second is from -drawlambda.

-bs-eval doesn't create intermediate file. Useful for learning or troubleshooting.

-bs-no-builtin-ppx

If you don't use any BS-specific annotations, you can explicitly turn it off. Another use-case is to use -ppx explicitly as below:

bsc -c -ppx bsppx.exe -bs-no-builtin-ppx c.ml

-bs-no-version-header

Don’t print BS version at the beginning of each JS file.

-bs-g (Experimental, since @3.1.0)

See Better Data Structures Printing (Debug Mode).
